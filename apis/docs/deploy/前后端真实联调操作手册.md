# 前后端真实联调操作手册

## 文档版本

- 版本：V1.0
- 更新时间：2026-04-26
- 维护人：Codex
- 文档定位：统一收口 `web + gateway + account-host + class-host + core-host + realtime-host` 的真实联调步骤、验收顺序、环境前置和排障口径。

## 1. 适用范围

本手册适用于以下真实链路联调：

- `web`
- `exam-gateway`
- `exam-account`
- `exam-class`
- `exam-core`
- `exam-realtime`

说明：

- 当前 `2C2G` 服务器收敛为 `5` 个后端进程。
- `exam-account` 聚合承接 `account + system`。
- `exam-class` 聚合承接 `class + question + paper + resource`。
- `exam-core` 聚合承接 `core + score + issue-core`。
- `exam-realtime` 聚合承接 `realtime + issue-notify`。
- 正式交卷结果仍由 `exam-core` 写入正式成绩表；问题主流程也由 `exam-core` 内部聚合承接。

## 2. 联调前置

### 2.1 SSH 隧道与中间件

如果当前 MySQL、Redis、Nacos 部署在远端服务器，必须先保持 SSH 隧道常驻。
推荐沿用 `docs/deploy/README.md` 中的隧道方式，确保以下端口本地可访问：

- `3306` MySQL
- `6379` Redis
- `8848` Nacos HTTP
- `9848` Nacos gRPC
- `9849` Nacos cluster/gRPC
- `15672` RabbitMQ 管理台

若隧道断开，常见现象是：

- `exam-realtime`、`exam-core` 启动失败或注册失效
- 网关访问 `/api/exam/realtime/**`、`/api/issue/notify/**` 返回 `503`

### 2.2 数据库脚本与联调种子

真实联调前，至少保证以下资产已经就绪：

- 全量建表脚本：`docs/sql/mysql.sql`
- 最小联调种子：`scripts/sql/live-integration-seed.sql`
- 冒烟脚本：`scripts/live-integration-smoke.ps1`
- 联调状态重置脚本：`scripts/reset-live-integration-state.ps1`
- `exam-realtime` 稳定启动脚本：`scripts/start-realtime.ps1`
- 联调一键准备脚本：`scripts/prepare-live-integration.ps1`
- 考试答题主链验收脚本：`scripts/verify-live-exam-flow.ps1`
- 问题通知主链验收脚本：`scripts/verify-live-issue-flow.ps1`
- 真实联调整体验收脚本：`scripts/verify-live-integration.ps1`

当前仓库 MySQL 初始化脚本约束：

- 共 `36` 张表
- `0` 个外键
- 仅保留主键、唯一键和普通索引

## 3. 启动顺序

真实联调固定按以下顺序进行：

1. 启动基础依赖：`MySQL -> Redis -> Nacos`
2. 启动入口服务：`exam-account -> exam-gateway`
3. 启动聚合业务宿主：`exam-class -> exam-core`
4. 启动实时服务：`exam-realtime`
5. 启动前端：`web`

本地默认端口：

- `8080` `exam-gateway`
- `8081` `exam-account(account + system)`
- `8082` `exam-class(class + question + paper + resource)`
- `8086` `exam-core(core + score + issue-core)`
- `8090` `exam-realtime(realtime + issue-notify)`
- `5173` `web`

Node 服务联调日志当前可查看：

- `.runtime-logs/exam-realtime.out.log`

## 5. 次级验收链路：问题通知主链

完成以下真实链路：

`create -> issueNotify -> handle -> processNotify -> track`

### 5.2 验收步骤

1. 学生创建一条考试类问题：

- `POST /api/issue/core/create`

2. 教师侧确认收到：

- `PUT /api/issue/core/handle`

- `processNotify`

- `POST /api/issue/core/track`

6. 确认轨迹中至少出现：

- `CREATED`
- `HANDLED`

## 6. 常见故障排查

### 6.1 网关返回 `503`

优先排查顺序：

1. 下游服务是否已监听端口
2. Nacos 是否注册成功
3. SSH 隧道是否还在
4. `exam-gateway` 服务发现配置是否正常

重点关注：

- `8080`
- `8081`
- `8086`
- `8090`

### 6.1.1 登录页可访问但登录后直接掉到 `403`

常见表现：

- 账号密码登录接口返回 `200`
- 页面却跳到 `/403`
- 学生明明已经登录成功，却看不到 `/student` 或答题页

原因：

- `exam-account` 未启动，导致 `/api/system/permission/query` 无法初始化角色权限
- 或 `exam-account` 聚合出来的权限模板与前端当前真实页面不一致

处理方式：

1. 确认 `exam-account` 已启动且网关可转发 `/api/system/**`
2. 直接检查 `/api/system/permission/query` 是否包含当前角色首页、隐藏业务页和对应按钮权限
3. 若浏览器保留了旧 token，先清理本地存储后再重新登录

### 6.1.2 答题页接口返回 `200`，浏览器却报 `AxiosError: Network Error`

常见表现：

- `/api/exam/realtime/session` 在脚本或 Postman 中返回 `200`
- 浏览器答题页仍显示 `00:00`、`0` 题
- 控制台出现 `AxiosError: Network Error`

原因：

- 网关和 Node 下游同时写入 CORS 响应头，导致 `Access-Control-Allow-Origin` 重复

处理方式：

1. 确认网关已启用 `DedupeResponseHeader=Access-Control-Allow-Origin Access-Control-Allow-Credentials, RETAIN_FIRST`
2. 重启 `exam-gateway`
3. 用浏览器或 `Invoke-WebRequest` 检查 `/api/exam/realtime/session` 的实际响应头，确认 `Access-Control-Allow-Origin` 只保留一份

### 6.2 数据库缺表

常见表现：

- `/api/exam/realtime/session` 提示 `exam_instance` 不存在
- `/api/issue/notify` 提示 `exam_issue_record` 不存在

处理方式：

1. 先执行 `docs/sql/mysql.sql`
2. 再执行 `scripts/sql/live-integration-seed.sql`
3. 最后重跑 `scripts/live-integration-smoke.ps1`

### 6.3 缺少业务种子

常见表现：

- 考试不存在
- 学生未分发
- 问题列表为空但并非接口故障

处理方式：

- 使用 `scripts/sql/live-integration-seed.sql` 补最小真实联调样例

### 6.4 Nacos 未注册

常见表现：

- 网关可启动，但实时链路仍返回 `503`
- 本地端口已监听，但网关 `lb://` 仍不可达

处理方式：

- 检查 `127.0.0.1:8848/nacos`
- 检查服务名是否存在：
  - `exam-core`
  - `exam-realtime`
  - `exam-account`

### 6.5 Socket 断连

常见表现：

- 答题页提示“实时连接已断开，正在尝试恢复”
- 问题页无法收到通知推送

处理方式：

1. 确认 `VITE_WS_BASE_URL`、`VITE_ISSUE_WS_BASE_URL`
2. 确认 Node 实时服务仍在运行
3. 确认 token 未失效
4. 确认前端当前为真实模式而不是 mock

### 6.6 学生已正式交卷

常见表现：

- `/api/exam/realtime/session` 返回“当前考试已正式提交”

处理方式：

- 不直接判定为代码错误
- 先检查当前考试样例是否已经被提交过
- 必要时重跑种子或补一份未提交考试样例

### 6.7 重复复用同一考试样例时提交被锁定

常见表现：

- 重跑种子后立刻再次验证 `/api/exam/realtime/submit`
- 返回“当前考试正在提交中”

原因：

- `exam-realtime` 使用 Redis 提交锁防止重复提交
- 如果连续复用同一组 `examId/studentId` 做多轮验证，旧锁在 TTL 未过期前仍可能存在

处理方式：

1. 先重跑 `scripts/sql/live-integration-seed.sql`
2. 再清理对应 Redis 联调键，或等待提交锁 TTL 自然过期
3. 然后重新执行答题主链验证

当前联调样例对应的 Redis 键粒度为：

- 草稿答案
- 活跃连接
- 提交锁

如果希望一次性完成“重跑种子 + 清理 Redis 联调键”，可直接执行：

```powershell
pwsh -File .\scripts\reset-live-integration-state.ps1
```

### 6.8 `exam-common` 变更后 `exam-core` 的问题主流程接口返回 `500`

常见表现：

- `create` 正常
- `handle`、`track` 返回 `500`
- 日志中出现 `NoSuchFieldError: DomainErrorCode ISSUE_NOT_FOUND`

原因：

- `exam-core` 运行进程仍加载旧版 `exam-common` 类定义

处理方式：

1. 重新构建 `exam-common` 与 `exam-core`
2. 彻底重启 `exam-core`
3. 若 `spring-boot:run` 场景下仍混入旧依赖，优先使用打包后的 `jar` 方式启动

推荐重建命令：

```powershell
.\mvnw.cmd -pl exam-core -am package -DskipTests
```

如果希望把“关键端口预检查 + 必要时补拉 realtime + 重置联调状态 + 运行冒烟”串成一步，可直接执行：

```powershell
pwsh -File .\scripts\prepare-live-integration.ps1 -EnsureRealtime
```

## 7. 执行优先级与完成标准

### P0 环境可用性校验

固定动作：

- 运行 `scripts/live-integration-smoke.ps1`
- 或直接运行 `scripts/prepare-live-integration.ps1 -EnsureRealtime`

完成标准：

- 端口可达
- 关键表齐全
- 最小种子可读
- 网关路由不再返回 `503`
- `/api/system/permission/query` 可正常返回当前角色权限模板

### P1 考试答题主链真实联调

固定动作：

- 以真实前后端链路验证 `/session -> enterExam -> save-progress -> reportScreen -> submit`

完成标准：

- 页面成功进入答题页
- 倒计时可持续接收
- 保存与异常上报可用
- 提交流程可落到 `exam-core`

### P2 问题通知链真实联调

固定动作：

- 验证 `create -> issueNotify -> handle -> processNotify -> track`

完成标准：

- 页面行为与后端接口口径一致
- 创建与处理通知到达正确角色
- 轨迹页顺序正确

### P3 环境稳定性收口

固定动作：

- 收口 SSH 隧道、Nacos 检查、日志位置、重启顺序

完成标准：

- 环境异常时有固定排查路径
- 同学接手时不需要重新摸索启动顺序
