# 前端刷新链路调试经验：从“刷新令牌失效”到“刷新后 404”

## 1. 文档目的

- 记录本次前端在浏览器整页刷新场景下的完整排查过程。
- 形成一份可复用的调试经验文档，便于后续定位“登录态恢复”和“动态路由恢复”类问题。
- 本文覆盖的问题链路包括：
  - 刷新业务页后控制台报错：`刷新令牌失效`
  - 修复登录态恢复后，刷新业务页继续跳转到 `404`

## 2. 问题背景

- 当前前端工程采用 `Vue3 + Vite + Pinia + Vue Router + Element Plus`。
- 在后端尚未联调完成前，前端主要依赖 `Mock` 数据闭环运行。
- 登录态采用：
  - `accessToken`
  - `refreshToken`
  - 用户信息持久化到浏览器本地存储
- 业务页路由采用“登录后按角色动态注册”的方案。

## 3. 第一阶段问题：刷新后提示“刷新令牌失效”

### 3.1 现场现象

- 用户登录后可以正常进入班级页、试卷页、考试页等业务页。
- 但在业务页执行浏览器整页刷新后，控制台出现如下错误：

```text
MockHttpError: 刷新令牌失效
    at fail (shared.ts:40:9)
    at handleAccountMock (account.ts:67:7)
    at handleMockRequest (index.ts:37:20)
    at async request (http.ts:83:25)
    at async Proxy.refreshSession (auth.ts:43:22)
    at async request (http.ts:104:15)
    at async Proxy.initialize (permission.ts:47:22)
    at async index.ts:92:5
```

### 3.2 初步判断

- 报错栈已经明确指出问题不在普通业务接口，而在“登录态刷新链路”。
- 关键链路是：
  - 页面刷新
  - 权限初始化
  - 请求返回 `401`
  - 自动触发 `refreshSession`
  - Mock 刷新令牌接口判定失败

### 3.3 排查路径

- 先读请求拦截与重试逻辑：
  - `web/src/api/http.ts`
- 再读登录态存储逻辑：
  - `web/src/stores/auth.ts`
  - `web/src/utils/auth.ts`
- 再读 Mock 刷新令牌逻辑：
  - `web/src/mock/handlers/account.ts`
  - `web/src/mock/data.ts`

### 3.4 根因定位

根因不是“本地没有 token”，而是 **Mock token 校验依赖纯内存映射表**。

- 前端会把以下信息持久化到本地存储：
  - `accessToken`
  - `refreshToken`
  - `user`
- 但 Mock 中用于校验 token 的两张表：
  - `accessTokenMap`
  - `refreshTokenMap`
  只存在于当前页面运行时内存中。

这会导致：

1. 登录成功时，token 被写入浏览器本地存储，同时也写入 Mock 内存映射表。
2. 浏览器整页刷新后，本地存储仍在，但 Mock 模块被重新加载，内存映射表清空。
3. 页面刷新后首次请求因为 `accessToken` 找不到映射而失效。
4. 自动触发 `refreshToken` 时，`refreshTokenMap` 同样为空，于是报出“刷新令牌失效”。

### 3.5 修复思路

要解决这个问题，有两个方向：

- 方向 A：
  - 把 Mock token 映射表也持久化到浏览器存储
- 方向 B：
  - 不再完全依赖映射表，而是允许 Mock 从 token 本身恢复出账号身份

本次选择 **方向 B**，原因是：

- 改动更小
- 不需要额外引入一套 Mock token 持久化协议
- 更适合当前本地开发和演示场景

### 3.6 具体修复

#### 修复点 1：新增 token 反解析能力

- 在 `web/src/mock/data.ts` 中新增 `resolveAccountIdFromToken`
- 允许从如下 token 中恢复出 `accountId`

```text
access-u-student-01-xxxxxxx
refresh-u-student-01-xxxxxxx
```

#### 修复点 2：访问令牌恢复用户时，不再只依赖内存表

- `getUserByToken` 逻辑改为：
  - 先查 `accessTokenMap`
  - 查不到时再从 token 字符串中解析 `accountId`
  - 解析成功后回填到 `accessTokenMap`

#### 修复点 3：刷新令牌接口支持整页刷新后的恢复

- `web/src/mock/handlers/account.ts` 中 `/api/account/refresh-token` 的逻辑改为：
  - 先查 `refreshTokenMap`
  - 查不到时再从 `refreshToken` 字符串中解析 `accountId`
  - 解析成功后回填到 `refreshTokenMap`

### 3.7 验证方式

为了避免“看起来修好了，实际上只是手动场景没复现”，补了专门的回归测试：

- `web/src/tests/http-refresh.spec.ts`

新增的关键验证点：

1. 访问令牌失效后仍能自动刷新并重试请求
2. **模拟整页刷新导致 Mock 内存 token 表被清空后**
   - 仍然可以通过 `refreshToken` 恢复权限初始化

### 3.8 阶段结论

- “刷新令牌失效”问题已修复
- 但在继续手工验证时，暴露出第二个问题：**业务页刷新后会跳到 404**

## 4. 第二阶段问题：刷新业务页后跳转 404

### 4.1 现场现象

- 修复登录态刷新后，再次测试浏览器整页刷新。
- 此时不再报“刷新令牌失效”，但会直接进入：
  - `/404`
  - 页面提示“页面不存在”

### 4.2 初步判断

- 这说明登录态恢复已经基本正常。
- 当前问题更像是：
  - 页面地址是对的
  - 但在刷新瞬间，Vue Router 还没有识别到这个业务路由

### 4.3 排查路径

重点检查三块：

- 路由静态注册和守卫逻辑：
  - `web/src/router/index.ts`
- 动态路由注册逻辑：
  - `web/src/stores/permission.ts`
- 路由注册源数据：
  - `web/src/router/route-registry.ts`

### 4.4 根因定位

根因是 **动态路由注册时机晚于浏览器首次路径匹配**。

当前工程的业务页路由并不是启动时全部注册，而是：

1. 用户登录
2. 拉取权限
3. 根据权限结果动态注册业务路由

这套方案平时导航没有问题，但整页刷新时会出现时序问题：

1. 浏览器直接访问当前业务页地址，例如 `/teacher/paper`
2. Vue Router 初始化时，静态路由表里还没有这个业务路由
3. 命中兜底路由 `/:pathMatch(.*)*`
4. 被重定向到 `/404`
5. 后续即使权限初始化再补路由，也已经晚了

### 4.5 修复思路

整页刷新要能恢复业务页，关键在于：

- **在首次路径匹配前，就让当前用户角色对应的业务路由可用**

注意这里不能简单把所有业务路由都静态注册，否则会削弱权限边界设计。

因此本次采用折中方案：

- 如果浏览器本地已经有登录用户
- 则在应用启动时，先按 `user.roleType` 预注册该角色可见的业务路由
- 后续仍然通过权限初始化和守卫判断是否真正允许进入

### 4.6 具体修复

#### 修复点 1：在路由层增加按角色预注册的纯函数

- 在 `web/src/router/index.ts` 中新增：

```text
registerRoutesForRole(role)
```

该函数负责：

- 根据角色从 `route-registry` 中拿到该角色应有的业务路由
- 转成 Vue Router 路由记录
- 在路由表中补注册

#### 修复点 2：应用启动时先根据本地用户角色补路由

- 在路由模块初始化阶段：
  - 先读取本地 `authStore`
  - 如果已有 `user`
  - 立即执行 `registerRoutesForRole(user.roleType)`

这样业务页在整页刷新时，不会因为“路由还没注册”先掉进 `404`

#### 修复点 3：守卫中继续兜底确保角色路由存在

- 在 `beforeEach` 中，对于已登录用户，再次调用：

```text
registerRoutesForRole(authStore.user.roleType)
```

这样可以覆盖：

- 首次启动恢复
- 后续会话切换
- 某些测试场景下手工移除动态路由后的再进入

### 4.7 处理中的一个中间坑

首次修复时，尝试把“预注册角色路由”写进 `permission store`。

这样做虽然思路没错，但在测试环境里触发了：

- `router`
- `permission store`
- `router`

之间的循环依赖风险，导致测试报错。

后续改成：

- 把“按角色注册路由”做成 `router` 模块里的纯函数
- `permission store` 只继续负责权限载荷与按权限注册

这样职责更清晰，也避免了初始化时的依赖环。

### 4.8 验证方式

新增并修正了路由回归测试：

- `web/src/tests/router-guard.spec.ts`

核心验证点：

1. 学生正常访问自己的业务页
2. **模拟整页刷新后**
   - 即使先清空动态注册结果
   - 仍然可以重新进入已有权限的业务页
   - 不会掉到 `/404`

## 6. 本次调试经验总结

### 6.1 经验一：Mock 登录态不能只依赖运行时内存

- 如果 token 被持久化到了浏览器
- 那么 Mock 校验依据也必须具备跨整页刷新的恢复能力
- 否则就会出现“本地 token 明明还在，但刷新后全部失效”的假问题

### 6.2 经验二：动态路由方案必须单独考虑整页刷新

- 路由懒注册只考虑“登录后跳转”是不够的
- 还必须覆盖：
  - 浏览器直接打开业务页地址
  - 浏览器整页刷新业务页
  - 会话恢复后的首次路径匹配

### 6.3 经验三：排查要分层，不要一开始就怀疑所有模块

这次比较有效的排查顺序是：

1. 先看报错栈属于哪条链路
2. 先修登录态恢复
3. 再看刷新后的路由恢复
4. 每修一段就补测试

这种“按链路层层剥离”的方式，比一开始同时改 token、权限、路由更稳。

### 6.4 经验四：修复后一定要补“更像真实用户操作”的测试

本次新增的两个关键测试，本质上就是在模拟真实用户行为：

- 刷新后 Mock 内存丢失
- 刷新后动态路由丢失

比单纯测“函数返回值对不对”更能提前挡住回归。

## 7. 后续建议

- 在进入真实后端联调前，继续保留这两类回归测试：
  - 登录态刷新恢复
  - 整页刷新后的业务路由恢复
- 后端联调后，再重点验证：
  - 真实 `refreshToken` 失效策略
  - 真实 401 重试边界
  - 真实菜单/权限载荷下的业务页直达与刷新场景

## 8. 一句话结论

这次问题不是单点 bug，而是一个典型的“**整页刷新场景下，Mock 登录态恢复 + 动态路由恢复顺序不完整**”的问题链。
修复后，前端已经能在 Mock 模式下稳定承接“登录 -> 进入业务页 -> 浏览器刷新 -> 保持当前页”的核心体验。
