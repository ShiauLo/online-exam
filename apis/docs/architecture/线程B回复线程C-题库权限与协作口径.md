## 一、线程 B 可确认的最终口径

### 1. 题库域数据范围

- 教师仅可操作本人创建的题目，以及个人分类。
- 管理员可查看和管理全局题目、全局分类以及审核流程。
- 审计员在题库域为只读角色，可按授权范围查询和导出，导出结果需脱敏。

说明：

- 教师“仅操作本人创建数据”的口径与角色文档保持一致。
- 管理员对题库域具备全局管理和审核职责。
- 审计员不参与题库数据增删改，只承担核查和只读导出职责。

### 2. 题库审核归属

- 题库审核归管理员端，不归教师端。
- 接口口径沿用：`/api/question/audit`
- 管理端页面口径沿用：`/admin/question-audit`

说明：

- 接口文档已明确 `/api/question/audit` 仅超级管理员和普通管理员可审核。
- 前端设计文档已明确“试题审核页”位于管理员端。

### 3. 教师端是否需要查询接口

- 教师端试题管理页明确需要列表查询能力。
- 因此线程 C 首版应补齐：`/api/question/query`

说明：

- 教师端页面已明确存在题目列表、分类筛选和状态筛选需求。
- 若缺少查询接口，则教师端试题管理页无法完整落地。

### 4. 题库权限编码风格

线程 B 建议线程 C 第一阶段统一采用小写点式编码：

- `question.query`
- `question.create`
- `question.update`
- `question.delete`
- `question.toggleStatus`
- `question.category.manage`
- `question.import`
- `question.export`
- `question.audit`

说明：

- 该风格更贴近前端动态权限的 `buttonList` 设计。
- 该风格也更利于后续 `exam-system /permission/query` 直接输出菜单、路由和按钮权限清单。

## 二、线程 C 可按此先行实现的默认假设

在线程 A 尚未完全冻结共享契约前，线程 C 可按以下假设先行推进：

- 教师仅管理本人题目与个人分类。
- 管理员管理全局题目、全局分类和审核流程。
- 审计员只读，导出结果按角色要求脱敏。
- 首版补齐 `/api/question/query`。
- 教师端页面路由按 `/teacher/question`。
- 管理员端审核页路由按 `/admin/question-audit`。
- 题库权限编码先按小写点式口径实现。

## 三、线程 A 仍需冻结的事项

以下内容不由线程 B 单独拍板，线程 C 如需依赖，仍应以线程 A 最终冻结口径为准：

- 统一响应结构是否下沉到 `exam-common`
- JWT claim 中 `userId / roleId` 的最终口径
- 网关透传 Header 的最终协议
- `/api/question/** -> lb://exam-question` 的网关路由接入
- 公共错误码与异常模型
- 角色命名与 `roleId` 映射的最终冻结口径

## 四、当前仓库可参考但尚未正式冻结的现状

基于当前仓库现状，`exam-account` 中已有如下角色映射：

- `roleId=2` -> `ADMIN`
- `roleId=3` -> `TEACHER`
- `roleId=5` -> `AUDITOR`

说明：

- 线程 C 可将其作为首版本地判断的参考现状。
- 该映射目前仍属于“现有实现口径”，不应被扩散为跨服务永久契约。
- 若线程 A 后续冻结出不同最终口径，应统一回收调整。

## 五、对线程 C 第一阶段的实施建议

线程 C 第一阶段建议优先落地：

1. 题目新增、编辑、删除、禁用/启用
2. 分类管理
3. 查询接口
4. 导入导出
5. 审核接口

实施时建议保持以下收敛原则：

- 教师端和管理员端共享题库基础模型，但数据范围严格区分。
- 审核能力只放管理员端，不给教师端开放。
- 查询接口优先按前端页面最小需求实现，不等待后续模块补齐。
- 权限编码保持单一风格，不采用大小写混合或双轨映射。

## 六、备注

- 本文档是线程 B 对线程 C 的业务协同回复，不替代线程 A 的共享基线冻结职责。
- 若线程 A 后续对角色编码、响应结构、JWT 或网关规则做正式收敛，线程 C 应以线程 A 最终口径为准。
- 若线程 C 后续对题库页面、权限码、审核流做结构性调整，应同步反馈线程 B，以便后续 `exam-system` 权限模板保持一致。
