# 未来微服务推进优先级与冻结标准

## 文档版本

- 版本：V1.4
- 更新时间：2026-04-26
- 维护人：Codex

## 1. 文档定位

- 目标：作为当前 5 个运行服务内后续能力域演进与历史拆分冻结的单一规划来源，统一承接推进顺序、冻结标准、测试冻结基线和首批基准。
- 适用范围：当前 5 个运行服务 `exam-gateway`、`exam-account`、`exam-class`、`exam-core`、`exam-realtime` 所承接的试卷、考试、成绩、资源、问题申报、实时交互、问题通知等能力域，以及这些能力域的历史拆分记录。
- 使用原则：本文保留完整历史推进链，但当前运行拓扑固定为 5 个运行服务；文中出现的 `exam-system`、`exam-question`、`exam-paper`、`exam-resource`、`exam-score`、`exam-issue-core`、`exam-issue-notify` 默认视为能力域或历史拆分名，不再表示当前独立运行微服务。

## 2. 推进顺序

1. 试卷能力域（历史拆分名：`exam-paper`，现由 `exam-class` 承接）
2. 考试主流程增强（运行服务：`exam-core`）
3. 成绩能力域（历史拆分名：`exam-score`，现由 `exam-core` 承接）
4. 资源能力域（历史拆分名：`exam-resource`，现由 `exam-class` 承接）
5. 问题申报能力域（历史拆分名：`exam-issue-core`，现由 `exam-core` 承接）
6. 实时考试能力域（运行服务：`exam-realtime`）
7. 问题通知能力（历史拆分名：`exam-issue-notify`，现由 `exam-realtime` 承接）

说明：

- 上述顺序保留完整历史推进链，但当前统一按能力域推进，不再按旧拆分名恢复独立运行微服务。
- 上述规划内的能力域当前已全部落地到 5 个运行服务中；若后续再新增真正的新服务，需在本文基础上重新冻结优先级、宿主边界和测试基线。

## 3. 排序依据

### 1. `exam-paper`

- 排位原因：最早承接现有 `exam-question`、`exam-class`、`exam-system` 的稳定产物，并直接验证题库 `referenceCount/referenceLocked` 是否足够支撑试卷域。
- 直接依赖：`exam-question`、`exam-class`、`exam-system`
- 阻塞关系：阻塞 `exam-core` 和 `exam-score`
- 启动前必须稳定的共享契约：`QuestionType`、`QuestionAuditStatus`、`ApiResponse`、`PageResult`、`AuditTrail`、题库引用锁定语义

### 2. `exam-core`

- 排位原因：考试实例、分发、状态流转都建立在试卷基线之上，必须等 `exam-paper` 稳定后再做。
- 直接依赖：`exam-paper`、`exam-class`、`exam-account`
- 阻塞关系：阻塞 `exam-score` 和 `exam-realtime`
- 启动前必须稳定的共享契约：共享请求头常量、统一 `401/403` 返回、班级成员状态口径、试卷状态口径

### 3. `exam-score`

- 排位原因：评分、发布、成绩分析依赖考试实例和试卷结构稳定，不适合先行。
- 直接依赖：`exam-core`、`exam-paper`
- 阻塞关系：不阻塞 `exam-resource`，但会影响后续成绩申诉和审计扩展
- 启动前必须稳定的共享契约：共享审计模型、题型枚举、统一错误码映射、考试完成态口径

### 4. `exam-resource`

- 排位原因：属于支撑型服务，重要但不应抢在考试主链路前面。
- 直接依赖：`exam-question`、`exam-paper`、`exam-score`
- 阻塞关系：不阻塞主链路启动，但会影响题图、模板和导出文件的正式化
- 启动前必须稳定的共享契约：统一下载响应、`X-Request-Id` 透传口径、资源授权边界

### 5. `exam-issue-core`

- 排位原因：问题申报是跨域流程服务，依赖考试/系统边界清晰后再做更稳。
- 直接依赖：`exam-account`、`exam-core`、`exam-system`
- 阻塞关系：阻塞 `exam-issue-notify`
- 启动前必须稳定的共享契约：共享审计字段标准、共享错误码、系统权限模板口径

### 6. `exam-realtime`

- 排位原因：实时交互必须建立在考试核心服务稳定之后，避免实时协议跟着业务反复改。
- 直接依赖：`exam-core`、`exam-account`
- 阻塞关系：不阻塞 `exam-issue-notify`，但会影响考试过程体验正式化
- 启动前必须稳定的共享契约：Token 解析口径、考试状态枚举来源、`X-Request-Id` 链路透传

### 7. `exam-issue-notify`

- 排位原因：纯通知服务，必须等问题申报主流程稳定后再启动，避免推送协议漂移。
- 直接依赖：`exam-issue-core`
- 阻塞关系：不阻塞考试主链路
- 启动前必须稳定的共享契约：问题状态流转枚举来源、共享通知审计字段标准

## 4. 冻结标准

### 1. 职责边界冻结

- 每个服务只负责一段主链路，不允许与现有已落地服务或其他未来服务重复承载相同业务职责。
- `exam-paper` 只负责组卷和试卷生命周期，不负责考试分发。
- `exam-core` 只负责考试实例和考试状态流转，不负责评分。
- `exam-score` 只负责评分、成绩发布和成绩分析，不负责考试过程控制。
- `exam-resource` 只负责资源分发与下载，不承载业务主数据。
- `exam-issue-core` 只负责问题申报流程，不承载实时推送。
- `exam-realtime` 只负责实时考试交互，不承载考试主数据持久化规则。
- 历史拆分能力 `exam-issue-notify` 只负责问题通知，不承载问题处理主流程；当前运行时由 `exam-realtime` 承接。

### 2. 依赖方向冻结

- 不允许循环依赖。
- 依赖顺序必须满足：`exam-question/exam-class/exam-system -> exam-paper -> exam-core -> exam-score`
- `exam-resource` 作为支撑侧依赖业务域，不反向被主链路依赖。
- `exam-issue-core` 独立挂接 `exam-account/exam-core/exam-system`，不反向驱动考试主链路。
- `exam-realtime` 依赖 `exam-core/exam-account`，不反向定义考试主链路规则。
- 历史拆分能力 `exam-issue-notify` 只依赖 `exam-issue-core`；当前以 `exam-realtime` 的内聚能力形态存在。

### 3. 共享契约冻结

- 所有新服务必须复用 `exam-common` 的 `ApiResponse`、`PageResult`、共享枚举、共享错误码和 `AuditTrail`。
- 不允许在新服务里重新定义本地 `Result`、`PageResult`、题型枚举、审核状态枚举或审计字段标准。
- 网关请求头口径统一沿用：
  - `Authorization`
  - `Request-Id`
  - `X-User-Id`
  - `X-Role-Id`
  - `X-Request-Id`
- 鉴权失败和权限失败继续沿用网关统一 `401/403` 结构，不新增新的公共 Header 或新的失败包装协议。

### 4. 错误码分段冻结

- `exam-paper`：`14xx`
- `exam-core`：`15xx`
- `exam-score`：`16xx`
- `exam-resource`：`17xx`
- `exam-issue-core`：`18xx`
- `exam-realtime`：`19xx`
- `exam-issue-notify`：`20xx`
- 新分段不得侵入现有 `10xx-13xx` 已落地范围。

### 5. 接口基线冻结

- 每个服务只冻结第一批必须落地的接口族，不提前扩展到第二阶段能力。
- 首批接口必须与总接口文档保持一致，不允许实现时再另起命名风格。
- WebSocket 事件只冻结事件名和最小职责，不在本轮提前扩展复杂消息协议。

### 6. 暂不实现边界冻结

- 已落地服务允许补齐对应网关路由；`exam-realtime` 当前已接入网关 `/api/exam/realtime/**` 与 `/api/issue/notify/**`。
- 当前仓库的运行拓扑固定为 `exam-gateway`、`exam-account`、`exam-class`、`exam-core`、`exam-realtime` 5 个服务；本文中的其余名称均按能力域或历史拆分记录理解。
- 不在本轮新增跨服务真实调用链。
- 不为未来服务提前发明本地共享层副本。
- 不把 `resource/realtime/notify` 的技术实现细节提前混入业务服务文档。

## 5. 冻结验证基线

- 职责唯一性校验：每个待落地服务的职责不能与现有已落地服务或其他未来服务重叠。
- 依赖方向校验：依赖关系必须满足推进顺序，且不存在循环依赖。
- 共享契约单一来源校验：分页、响应体、共享枚举、共享审计、共享错误码都必须引用 `exam-common` 单一来源。
- 错误码分段不冲突校验：新服务编号段不能与 `DomainErrorCode` 当前已落地段冲突。
- 接口一致性校验：每个服务的首批接口族必须与总接口文档一致。
- 可执行性校验：仅凭本文与总接口文档，后续实现者无需再补问边界即可开始实现首个服务。

## 6. 各微服务基准

### 1. `exam-paper`

- 最小职责：手动组卷、自动组卷、试卷编辑、审核、发布、回收
- 当前现状：已完成 Maven 模块、Spring Boot 应用入口、基础配置、请求上下文解析、统一异常处理、数据库表结构、查询与首批接口实现、服务测试与网关路由；跨服务联调尚未开始。
- 首批接口：
  - `/api/paper/create/manual`
  - `/api/paper/create/auto`
  - `/api/paper/update`
  - `/api/paper/delete`
  - `/api/paper/audit`
  - `/api/paper/publish`
  - `/api/paper/terminate`
  - `/api/paper/recycle`
  - `/api/paper/export`
- 共享依赖：`QuestionType`、`QuestionAuditStatus`、`PageResult`、`AuditTrail`、题库 `referenceCount/referenceLocked`
- 错误码分段：`14xx`

### 2. `exam-core`

- 最小职责：考试创建、参数调整、分发、状态流转、补考审核
- 当前现状：已完成 Maven 模块、Spring Boot 应用入口、基础配置、请求上下文解析、统一异常处理、最小启动测试，以及考试创建、考试查询、考试参数调整、考试分发、考试暂停恢复、补考审核接口；首批接口已全部落地。
- 首批接口：
  - `/api/exam/core/create`
  - `/api/exam/core/query`
  - `/api/exam/core/update/params`
  - `/api/exam/core/distribute`
  - `/api/exam/core/toggle-status`
  - `/api/exam/core/approve-retest`
- 共享依赖：共享请求头常量、统一 `401/403` 返回、班级成员状态口径
- 错误码分段：`15xx`

### 3. `exam-score`

- 最小职责：自动评分、人工阅卷、成绩发布、成绩分析、申诉处理
- 首批接口：

  - `/api/score/manual-score`
  - `/api/score/publish`
  - `/api/score/query`
  - `/api/score/detail`
  - `/api/score/analyze`
- 共享依赖：共享审计模型、统一错误码映射、题型与审核状态共享枚举
- 错误码分段：`16xx`

### 4. `exam-resource`

- 最小职责：题图、模板、导出文件分发
- 当前现状：已完成 Maven 模块、Spring Boot 应用入口、基础配置模板、请求上下文解析、统一异常处理、网关路由、最小启动测试和首批资源接口；题库/试卷/成绩/系统日志导出已接入共享本地资源目录分发链路，当前仍保持本地文件方案，不扩到对象存储。
- 首批接口：
  - `/api/resource/question/img`
  - `/api/resource/paper/template`
  - `/api/resource/file/download`
- 共享依赖：统一下载响应、`X-Request-Id` 透传
- 错误码分段：`17xx`

### 5. `exam-issue-core`

- 最小职责：问题申报、处理、转派、关闭、跟踪

### 6. `exam-realtime`

- 错误码分段：`19xx`

### 7. `exam-issue-notify`

- 最小职责：问题申报相关实时通知

- 独立收口文档：`docs/architecture/exam-issue-notify技术栈与首批落地方案.md`
- 首批接口与事件：

  - WebSocket `connected`
  - WebSocket `issueNotify`

- 共享依赖：问题状态流转枚举来源、共享通知审计字段标准
- 错误码分段：`20xx`

## 7. 当前不实现边界

- 不在本轮继续扩展除已建基座外其他新微服务模块实现。
- 问题通知能力当前不新增数据库表；`exam-realtime` 当前只新增最小异常记录表 `exam_realtime_abnormal_record`，不扩展正式考试主链表。
- 不在本轮扩展真实跨服务调用。
- 不在本轮提前细化资源存储协议、实时消息负载细节或通知通道实现。

## 8. 与现有已落地服务的关系

- `exam-paper` 直接消费 `exam-question`、`exam-class`、`exam-system` 的稳定产物。
- `exam-core` 建立在 `exam-paper`、`exam-class`、`exam-account` 稳定口径之上。
- `exam-score` 建立在 `exam-core` 与 `exam-paper` 的正式模型之上。
- `exam-resource` 为题库、试卷、成绩提供资源分发支撑，但不反向定义业务边界。
- `exam-issue-core` 与 `exam-system`、`exam-core` 协同，但不替代任何既有权限或考试主流程职责。
- `exam-realtime` 与 `exam-core` 配合，负责过程交互而非主业务规则。
- `exam-realtime` 内聚的 issue-notify 能力仅补充 `exam-issue-core` 的通知链路，不反向承载问题主流程。
