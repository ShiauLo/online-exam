## 一、线程 C 后续需要继续做的事情

### 1. 题库域从内存实现切换到正式持久化

- 将当前内存版 `QuestionStore` 替换为数据库持久化实现。
- 落正式表结构，至少覆盖：
  - `question_item`
  - `question_category`
  - `question_audit_log`
- 将 `referenceCount` 从内存字段过渡为可持久化字段，并为后续 `paper` 服务接入预留维护机制。

### 2. 文档与 SQL 回补

- 把 `/api/question/query`、`/api/question/category/query` 正式补回接口文档。
- 在 `apis/docs/sql/mysql.sql` 中补齐题库域表结构和必要索引。
- 将首版实际权限码、角色边界、导入导出格式与现有架构文档保持一致。

### 3. 导入导出能力继续完善

- 将当前首版 CSV 导入导出能力扩展为正式模板格式。
- 增加导入失败明细、行级错误提示和更完整的内容校验。
- 评估后续是否需要异步导入任务和文件落资源服务。

### 4. 题库域规则继续细化

- 补齐教师创建“全局分类题目需审核”的更完整流转规则。
- 增加分类层级、禁用分类、删除分类约束等边界处理。
- 继续完善审计员导出脱敏规则，确保答案、解析等敏感字段口径稳定。

### 5. 为后续 `paper` 域做前置准备

- 冻结题型、审核状态、查询字段、导出字段等会被 `paper` 依赖的题库契约。
- 预留“已被试卷引用”查询能力，便于后续 `paper` 域接入真实引用关系。
- 准备题库查询能力给 `paper` 自动/手动组卷复用。

## 二、需要线程 A 完成的事项

### 1. 共享基线收敛

- 将统一响应结构正式下沉到 `exam-common`。
- 将鉴权 Header 常量、`requestId` 工具、题型枚举、审核状态枚举、分页对象逐步收敛到 `exam-common`。
- 统一题库域共享错误码口径，避免各服务继续各自定义。

### 2. 网关与鉴权正式落仓

- 确认并稳定维护 `/api/question/** -> lb://exam-question` 路由。
- 继续稳定 `X-User-Id`、`X-Role-Id`、`X-Request-Id` 透传方案。
- 保证后续网关实现与线程 A 冻结口径一致，不回退到旧 Header/旧响应模型。

### 3. 契约回收

- 在题库域协议稳定后，吸收线程 C 当前临时本地实现中的公共类型。
- 为后续 `class`、`system`、`paper`、`exam-core` 使用题库相关公共契约做统一基线收敛。

## 三、需要线程 B 完成的事项

### 1. 权限与页面联动

- 在 `exam-system` 中正式接入题库权限码：
  - `question.query`
  - `question.create`
  - `question.update`
  - `question.delete`
  - `question.toggleStatus`
  - `question.category.manage`
  - `question.import`
  - `question.export`
  - `question.audit`
- 将教师端 `/teacher/question` 与管理员端 `/admin/question-audit` 的菜单、路由、按钮权限配置落仓。

### 2. 角色与数据范围协同

- 在权限查询和菜单渲染中固定教师/管理员/审计员在题库域的数据范围定义。
- 确保系统权限模板与线程 C 当前实现口径一致：
  - 教师仅管理本人题目与个人分类
  - 管理员管理全局题目、全局分类与审核流程
  - 审计员只读查询与脱敏导出

### 3. 后台协作

- 为管理员端题库审核页补齐正式权限模板与页面显隐口径。
- 为教师端试题管理页继续明确最小联调字段，避免后续前后端口径漂移。

## 四、后续需要其他业务线程配合的事项

- 接入题库引用关系，正式维护 `referenceCount` 或替代引用约束。
- 复用题库查询能力，明确手动组卷与自动组卷对题目字段的最小依赖集。
- 明确题库审核通过范围与组卷可用范围之间的关系。
