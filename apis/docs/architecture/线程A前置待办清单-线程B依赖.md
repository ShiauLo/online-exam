# 线程 A 前置待办清单（线程 B：exam-class / exam-system 依赖）

## 文档版本

- 版本：V1.1
- 修订时间：2026-04-03
- 修订人：线程 B

## 文档定位

- 设计目标：明确线程 B 在推进 `exam-class` 与 `exam-system` 之前，对线程 A 的前置依赖项，以及对线程 C 的协同依赖项，避免并行开发期间出现共享协议漂移、重复实现和联调返工。
- 适用范围：当前 `apis` 仓库中的共享基线线程、基础业务线程和题库线程协作。
- 当前边界：本文档仅约束线程 A 需先冻结的公共契约、鉴权上下文和接入规则，以及线程 C 需协同对齐的权限模板与页面标识，不扩展到线程 B 内部实现细节。

## 线程关系说明

### 线程 A：共享基线线程

负责范围：

- `exam-common`
- `exam-gateway`
- `exam-account`
- 公共异常模型
- 统一返回结构
- 鉴权规则
- 通用 DTO / 枚举 / Header 约定

### 线程 B：基础业务线程

负责范围：

- `exam-class`
- `exam-system`

说明：

- 线程 B 将严格依赖线程 A 冻结后的共享契约推进。
- 在线程 A 未冻结相关基线前，线程 B 不应自行扩散公共协议定义。

### 线程 C：题库线程

负责范围：

- `question`

说明：

- 线程 C 不属于线程 B 首轮硬阻塞依赖。
- 但 `exam-system` 中的权限模板、菜单渲染和按钮权限标识，需要与线程 C 保持协同一致。

## 一、必须由线程 A 先完成的事项

### 1. 冻结统一返回体到 `exam-common`

当前统一返回结构仍主要存在于 `exam-account` 内部，线程 B 若直接推进，容易在 `exam-class` 与 `exam-system` 中重复定义一套返回模型。

线程 A 需要提供：

- 统一响应模型，固定字段为：
  - `code`
  - `msg`
  - `data`
  - `requestId`
  - `timestamp`
  - `errors`
- 成功与失败的统一构造方式
- `400 / 401 / 403 / 404 / 409 / 500` 的标准响应口径
- 统一的 message 文案风格

交付目标：

- 后续所有新服务直接复用 `exam-common` 的公共返回体
- 不再在各服务内部复制 `Result` 实现

### 2. 冻结 `class/system` 公共错误码与异常口径

线程 B 会直接依赖一批稳定错误码和异常类型。若该部分未冻结，后续接口实现、测试用例和联调行为会频繁变化。

线程 A 需要补齐至少以下场景：

#### 班级服务相关

- 班级不存在
- 班级码无效
- 教师建班数量超限
- 学生入班数量超限
- 学生已在班级中
- 学生存在待审批申请
- 强制班级不可自主退班
- 教师无权操作非本人班级
- 删除班级前仍存在已加入学生
- 入班审批状态冲突

#### 系统服务相关

- 角色不存在
- 权限不存在
- 权限分配越权
- 配置项不存在
- 配置修改需要二次验证
- 日志导出需要审批
- 数据备份失败
- 数据恢复失败
- 备份/恢复状态冲突
- 临时授权已过期或无效

交付目标：

- 统一错误码
- 统一异常类型
- 统一异常到 HTTP 状态码的映射方式

### 3. 冻结鉴权上下文透传协议

`exam-class` 与 `exam-system` 的大多数接口都依赖“角色 + 资源 + 操作”的三重校验，因此必须先明确网关到业务服务的透传边界。

线程 A 需要确认：

- 网关必传请求头：
  - `X-User-Id`
  - `Request-Id`
- 是否透传角色信息
  - 若透传，需要明确字段名与格式
  - 若不透传，需要明确服务侧通过数据库查询角色为统一方案
- 是否统一由网关处理 token 合法性校验
- 业务服务是否允许再次解析 token
- `401` 与 `403` 的统一返回格式

交付目标：

- 新服务不重复发散鉴权方案
- 所有服务共享一致的用户上下文获取方式

### 4. 冻结网关路由接入规则

线程 B 后续需要落地两个新服务，网关路由前缀和鉴权策略需先稳定。

线程 A 需要确认：

- 路由前缀：
  - `/api/class/**`
  - `/api/system/**`
- 服务命名：
  - `exam-class`
  - `exam-system`
- 是否存在白名单接口
  - 若有，需明确具体路径
  - 若无，则默认全部鉴权
- 错误响应是否由网关统一包装

交付目标：

- 避免线程 B 在服务完成后因路由命名或鉴权策略调整返工

### 5. 确认根工程可接入新模块

线程 B 需要将新服务纳入聚合工程构建与测试，因此根工程接入方式必须稳定。

线程 A 需要确认：

- 根工程允许新增模块：
  - `exam-class`
  - `exam-system`
- Maven 聚合方式不再临时调整
- 公共依赖接入方式沿用现有 Spring Boot 微服务模式

交付目标：

- 线程 B 可按现有工程规范直接新增模块并接入构建

## 二、最好由线程 A 同步完成的事项

### 1. 统一异常处理模板

建议线程 A 将现有异常处理模式整理为共享模板，供新服务直接复用。

建议统一内容：

- 参数校验异常处理
- 业务异常处理
- 未知异常处理
- `requestId` 获取方式
- 开发 / 生产环境错误信息暴露策略

### 2. 冻结角色命名口径

当前文档中角色值主要为：

- `student`
- `teacher`
- `admin`
- `super_admin`
- `auditor`
- `ops`

但现有代码和 SQL 中已存在大写命名风格。

线程 A 需要明确：

- 对外接口返回值采用哪套命名
- 服务内部枚举采用哪套命名
- 数据库存储采用哪套命名
- 是否允许“内部大写、接口小写”映射方案

交付目标：

- 避免 `exam-system /permission/query` 与权限判断口径不一致

### 3. 冻结审计留痕最小字段

`exam-system` 的高危接口与 `exam-class` 的部分管理操作都需要统一记录审计信息。

建议线程 A 明确最小字段：

- `requestId`
- `operatorId`
- `operatorRole`
- `ip`
- `target`
- `action`
- `result`
- `approver`
- `timestamp`

### 4. 冻结分页查询公共口径

建议线程 A 明确以下字段的类型、默认值与边界校验规则：

- `pageNum`
- `pageSize`

交付目标：

- 避免各服务自行解释分页参数

## 三、与线程 C 的协同依赖（非首轮硬阻塞）

### 1. 权限编码与模块命名协同

线程 C 后续需要确认：

- `question` 模块的权限编码命名规则
- 题目新增、编辑、删除、禁用、导入、导出、审核等操作的权限标识
- 模块名称、菜单名称和权限分组名称

建议至少统一以下权限标识：

- `question.create`
- `question.update`
- `question.delete`
- `question.toggleStatus`
- `question.import`
- `question.export`
- `question.audit`
- `question.category.manage`

交付目标：

- `exam-system` 的角色权限分配与权限查询接口能稳定输出题库模块相关权限
- 避免按钮权限码与前端、题库服务实现不一致

### 2. 前端菜单与路由标识协同

线程 C 后续需要确认：

- 教师端题库页面路由
- 管理员端题库审核页面路由
- 题库相关页面组件标识
- 页面按钮权限标识

建议至少冻结以下内容：

- 教师端菜单与路由：
  - 菜单：`question`
  - 路由：`/teacher/question`
- 管理员端菜单与路由：
  - 菜单：`question-audit`
  - 路由：`/admin/question-audit`

交付目标：

- `exam-system /permission/query` 返回的 `menuList / routeList / buttonList` 能稳定覆盖题库页面
- 避免线程 B 在系统权限模板中使用的路由和线程 C 实际落地页面不一致

### 3. 题库相关权限模板预留

线程 B 在实现 `exam-system` 时，会先提供系统权限查询和角色权限分配能力。

因此建议线程 C 协同确认：

- 题库模块是否需要区分教师个人题库权限与管理员全局审核权限
- 是否需要独立的“题库审核”权限组
- 题库分类管理是否单独作为权限项

交付目标：

- 线程 B 在 `exam-system` 中预置的权限模板能覆盖线程 C 的后续落地需要
- 避免后续再调整角色模板、菜单模板和按钮权限结构

### 4. 协同方式说明

- 线程 B 首轮实现时，可先按现有接口文档和前端设计文档预留题库模块相关菜单、路由和按钮权限模板。
- 若线程 C 尚未冻结最终命名，则线程 B 使用当前文档口径作为临时标准。
- 一旦线程 C 对题库权限码、路由或页面标识做正式冻结，应同步回收并对齐 `exam-system` 中的权限模板定义。

## 四、线程依赖优先级说明

当前线程 B 的依赖优先级如下：

1. 线程 A：硬阻塞依赖，必须先冻结共享契约后线程 B 才继续正式落地
2. 线程 C：协同依赖，不阻塞线程 B 首轮开工，但应尽早对齐权限模板与页面标识

## 五、线程 B 的开工条件

线程 B 在满足以下条件后即可继续推进：

1. 统一返回体已冻结
2. `class/system` 公共错误码已冻结
3. 网关透传头与鉴权边界已冻结
4. `/api/class/**` 与 `/api/system/**` 路由口径已冻结

补充说明：

- 线程 C 未完成时，线程 B 仍可先推进 `exam-class` 和 `exam-system` 核心实现。
- 线程 C 相关部分以“权限模板预留”和“命名协同”为主，不作为首轮硬阻塞项。

## 六、可以后补的事项

以下内容不是线程 B 首轮推进的强阻塞项，可在骨架稳定后再统一下沉：

- 通用分页 DTO / VO 下沉到 `exam-common`
- 通用审计日志抽象下沉到 `exam-common`
- 统一导入导出文件规范
- 统一备份恢复适配接口
- 更完整的菜单 / 路由 / 按钮权限模板

## 七、线程 A 完成后线程 B 的继续工作

在线程 A 完成以上事项后，线程 B 将继续推进：

- `exam-class` 模块结构搭建
- `exam-class` 表结构、DTO、Service、Controller、最小测试
- `exam-system` 模块结构搭建
- `exam-system` 权限查询、角色分配、配置 / 告警 / 日志 / 备份接口骨架与最小测试

## 八、备注

- 本清单优先级高于线程 B 的具体实现节奏。
- 若接口文档、角色文档与现有代码冲突，以线程 A 冻结后的共享口径为准。
- 线程 B 不负责发散式修改 `exam-common`，如需新增公共契约，应由线程 A 统一收敛后通知线程 B。
- 线程 C 的相关依赖以协同对齐为主，不改变线程 B 当前首轮交付目标。
