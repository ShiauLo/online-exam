# exam-core 收尾后新微服务启动优先级建议

## 文档版本

- 版本：V1.0
- 更新时间：2026-04-19
- 维护人：Codex

## 1. 文档定位

- 目标：在 `exam-core` 首批接口全部落地后，给出下一阶段“如果现在要新开微服务，应该先开什么”的执行级建议。
- 适用范围：当前 `apis` 仓库后续准备启动的新微服务。
- 使用原则：优先级以“是否直接承接当前已完成能力”“是否能尽快形成跨服务闭环”“是否会反向阻塞更多后续服务”为准。

## 2. 当前结论

如果现在开始新开下一个微服务，建议优先级如下：

1. `exam-score`
2. `exam-resource`
3. `exam-issue-core`
4. `exam-realtime`
5. `exam-issue-notify`

补充说明：

- 当前 `exam-score` 已经启动模块基座并落了查询/详情首批接口。
- 如果现在讨论“`exam-score` 之后下一个该开什么”，顺位自动前移为：`exam-resource -> exam-issue-core -> exam-realtime -> exam-issue-notify`。

## 3. 为什么这样排

### 1. `exam-score`

- 排第一的原因：`exam-paper + exam-core` 现在都已经形成首批闭环，成绩域已经具备启动条件。
- 直接收益：可以尽快把“考试完成后如何评分、发布、查询成绩”接起来，形成最重要的主业务闭环。
- 对后续的带动：`exam-score` 做完后，成绩查询、成绩明细、成绩分析这些前后端主链路都会被打通。
- 当前依赖成熟度：最高，几乎只依赖现在已经冻结的试卷和考试模型。

### 2. `exam-resource`

- 排第二的原因：资源服务是支撑型服务，技术边界清晰，容易和现有导出、模板、题图需求形成配合。
- 直接收益：可以把后续题图、模板、导出文件下载从业务服务里抽出来，避免继续在业务服务内混文件职责。
- 不排第一的原因：它不是主业务强阻塞项，晚于 `exam-score` 更合理。

### 3. `exam-issue-core`

- 排第三的原因：问题申报流程已经有比较完整的文档边界，但它对考试主链路不是第一优先级。
- 直接收益：能把业务类问题、考试类问题、系统类问题的处理流程正式化。
- 不再往前提的原因：当前最核心的考试主链路还差成绩域闭环，问题申报可以稍后。

### 4. `exam-realtime`

- 排第四的原因：实时服务对考试体验很重要，但它最依赖稳定的考试主链路，不适合抢在成绩域和资源域前面。
- 风险点：如果过早启动，实时协议、进入考试口径、提交流程很容易随着考试主链路变化而返工。

### 5. `exam-issue-notify`

- 排第五的原因：它本质上依赖 `exam-issue-core` 的主流程稳定后再做才合适。
- 当前定位：纯补充型服务，不应该先于问题申报主服务启动。

## 4. 建议的启动策略

### 方案 A：只开一个新微服务

- 直接开 `exam-score`。

### 方案 B：允许一主一辅并行

- 主线开 `exam-score`。
- 辅线开 `exam-resource`。

### 方案 C：继续严格单线程推进

- 先完成 `exam-score` 首批接口。
- `exam-score` 基线稳定后，再开 `exam-resource`。

## 5. 对每个候选服务的开工前检查

### `exam-score`

- `exam-paper` 状态与题型口径不能再漂移。
- `exam-core` 的考试实例、分发、暂停恢复口径保持稳定。
- 成绩域错误码段固定使用 `16xx`。

### `exam-resource`

- 需要先冻结下载鉴权、文件命名和导出文件归属规则。
- 只承接资源分发，不反向定义业务主数据。

### `exam-issue-core`

- 需要继续沿用现有共享审计字段和统一响应结构。
- 问题类型与权限边界要继续和角色文档一致。

### `exam-realtime`

- 必须直接复用 `exam-core` 的考试状态口径。
- 不在实时服务里重新定义考试主数据规则。

### `exam-issue-notify`

- 必须等待 `exam-issue-core` 主流程冻结后再开。
- 只负责通知，不承载问题处理主流程。
