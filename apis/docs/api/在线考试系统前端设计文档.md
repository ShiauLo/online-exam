# 在线考试系统前端设计文档（V2.3）

## 文档版本

- 版本：V2.3
- 修订时间：2026-04-26
- 修订人：Codex

## 文档定位

- 设计目标：定义本期前端页面结构、权限渲染、交互规则和接口映射，作为前端开发与联调依据。
- 适用范围：学生端、教师端、普通管理员端、超级管理员端、审计员端、系统运维端的 PC 后台系统。
- 版本边界：本版覆盖本期已确认业务范围，不包含家长端、小程序端和离线考试客户端。
- 本期角色范围：`学生`、`教师`、`普通管理员`、`超级管理员`、`审计员`、`系统运维`。
- 非功能目标：满足动态权限渲染、实时考试交互、统一异常处理和生产环境安全要求。
- 冲突处理原则：`角色权限以角色文档为准，接口定义以接口文档为准，页面交互以前端文档为准`。

## 二、项目概述

### 2.1 项目定位

本系统前端采用前后端分离架构，以 `Vue3 + Vite` 构建统一管理后台。权限拆分为 6 个角色，是为了同时满足最小权限控制、审计分离和系统运维独立管理的要求。

### 2.2 本期实现

1. 统一登录与权限初始化。
2. 六角色 Dashboard 与菜单动态渲染。
3. 成绩查询与问题申报主流程已落地，考试实时交互首批闭环已由 `exam-realtime` 承接。
4. 审计员核查视图与系统运维视图。
5. `exam-realtime` 的独立技术与联调口径统一收口在 `docs/architecture/exam-realtime技术栈与首批落地方案.md`。

### 2.3 当前运行服务与联调口径

本期前后端联调统一以 `5` 个运行服务为准，不再把历史拆分模块当作当前独立进程。

| 运行服务 | 本地端口 | 前端联调口径 | 当前承载范围 |
| --- | --- | --- | --- |
| `exam-gateway` | `8080` | 所有非实时 HTTP 请求默认走 `http://127.0.0.1:8080` | 网关统一入口、鉴权、转发 |
| `exam-account` | `8081` | 仅用于单服务排查或断点调试 | 账户、认证、系统配置、日志、备份恢复 |
| `exam-class` | `8082` | 仅用于单服务排查或断点调试 | 班级、试题、试卷、资源接口 |
| `exam-core` | `8086` | 仅用于单服务排查或断点调试 | 考试主流程、成绩、问题申报主流程 |
| `exam-realtime` | `8090` | WebSocket 与实时 HTTP 直连 `http://127.0.0.1:8090` | 考试实时交互、问题通知 |

1. 非实时接口统一通过 `exam-gateway` 暴露，前端不直接感知 `exam-account`、`exam-class`、`exam-core` 的内部地址。
2. 实时考试与问题通知统一由 `exam-realtime` 承接，按不同 `path` 区分能力。
3. 历史说明：旧文档中的 `exam-system`、`exam-score`、`exam-resource`、`exam-issue-core`、`exam-issue-notify` 仅表示历史模块名或接口归类，不再代表当前独立运行服务。

### 2.4 扩展设计

1. 家长端、小程序端、移动监考端。
2. 多租户主题与租户级权限隔离。
3. 更强的防作弊能力，如摄像头监考、人脸识别、双设备检测。

## 四、权限渲染核心逻辑

### 4.1 权限数据结构

登录成功后调用 `/api/system/permission/query` 获取当前用户权限清单。前端统一按 `菜单 + 路由 + 按钮` 三层渲染，不再使用静态角色写死页面能力。

```json
{
  "roleType": "teacher",
  "menuList": [
    { "menuId": "class", "menuName": "我的班级", "path": "/teacher/class", "icon": "School" }
  ],
  "routeList": [
    { "path": "/teacher/class", "component": "TeacherClassPage" }
  ],
  "buttonList": [
    "class.approveJoin",
    "class.removeStudent",
    "issue.transfer"
  ]
}
```

### 4.2 渲染流程

1. 登录成功后保存 `accessToken` 和 `refreshToken`，不保存明文密码。
2. 调用 `/api/system/permission/query` 获取权限清单并写入 Pinia。
3. 根据 `menuList` 动态生成侧边菜单，根据 `routeList` 动态注册路由。
4. 页面内按钮按 `buttonList` 控制显示与禁用。
5. 发生 `401` 时先尝试刷新登录态，失败后回到登录页。

### 4.3 安全约束

1. 前端统一使用 HTTPS 请求非实时接口，使用 WSS 建立实时连接。
2. 不使用前端密码伪加密口径，密码仅通过 HTTPS 传输并由服务端加密存储。
3. “记住登录状态”只保存 token，不保存账号明文密码。

## 六、通用交互规范

### 6.1 弹窗规范

1. 删除、终止、恢复、关闭问题等关键操作使用 `ElMessageBox.confirm`。
2. 新增和编辑操作使用 `ElDialog`，提交按钮在请求中显示 loading。
3. 实时通知使用 `ElNotification`，右上角弹出，可手动关闭。

### 6.2 表格规范

1. 分页默认 `pageNum=1`、`pageSize=10`。
2. 支持关键字搜索、状态筛选、时间筛选。
3. 危险操作按钮使用红色，查看类操作使用普通按钮或文字按钮。

### 6.3 异常处理规范

1. `400`：展示字段级错误信息。
2. `401`：尝试刷新登录态，失败后跳转登录页。
3. `403`：提示“无操作权限，请联系管理员”并可跳转 403 页面。
4. `404`：提示资源不存在或页面已失效。
5. `409`：展示具体业务冲突原因。
6. `500`：展示请求 ID，提示联系管理员或运维排查。

### 6.4 实时交互规范

1. 考试答题与切屏告警当前通过 `exam-realtime` 的 WebSocket 实时同步；问题通知当前也通过 `exam-realtime` 的 WebSocket 实时同步，但固定使用 `path=/issue-socket.io`，问题列表与轨迹仍通过 `/api/issue/core/query` 与 `/api/issue/core/track` 刷新。
2. 实时连接断开时提示用户自动重连状态。
3. 考试页面重连后需恢复倒计时、题目状态和已保存答案。
4. 若实时服务边界、事件或联调方式调整，优先以 `docs/architecture/exam-realtime技术栈与首批落地方案.md` 为主。
5. 问题通知建议单独配置 `VITE_ISSUE_WS_BASE_URL` 与 `VITE_ISSUE_WS_PATH=/issue-socket.io`；未单独配置地址时回退到 `VITE_WS_BASE_URL`。

## 七、部署规范

### 7.1 环境划分

| 环境 | 前端地址 | 网关地址（`exam-gateway`） | 实时服务地址（`exam-realtime`） |
| --- | --- | --- | --- |
| 开发环境 | `http://localhost:5173` | `https://dev-api.exam-system.com` | `wss://dev-realtime.exam-system.com` |
| 测试环境 | `https://test.exam-system.com` | `https://test-api.exam-system.com` | `wss://test-realtime.exam-system.com` |
| 生产环境 | `https://exam-system.com` | `https://api.exam-system.com` | `wss://realtime.exam-system.com` |

说明：

1. 线上与测试环境对前端只暴露 `exam-gateway` 与 `exam-realtime` 两类入口地址。
2. `exam-account`、`exam-class`、`exam-core` 作为网关后的聚合运行服务存在，不单独对浏览器暴露环境域名。

### 7.2 打包与安全

1. 生产构建使用 `vite build --mode production`。
2. 所有静态资源通过 HTTPS 分发，导出文件统一通过资源服务下载。
3. 前端仅保存登录态 token，不保存明文密码或“记住密码”类敏感数据。
