# 在线考试系统 - 角色与权限规范文档（V3.0）

## 文档版本

- 版本：V3.1
- 修订时间：2026-04-22
- 修订人：Maghert

## 文档定位

- 设计目标：为在线考试系统提供统一的角色、权限、审批链和数据范围定义，支撑后端鉴权、前端显隐、测试验收和答辩说明。
- 适用范围：学校或企业内部在线考试系统，适用于班级化教学考试、阶段测试、统一考试和成绩管理场景。
- 版本边界：本版本覆盖本期已确认的 6 角色权限模型、核心业务模块和生产环境安全约束，不包含家长端、多租户独立部署和应急旁路工具实现。
- 本期角色范围：`学生`、`教师`、`普通管理员`、`超级管理员`、`审计员`、`系统运维`。
- 非功能目标：满足最小权限、审计留痕、数据隔离、权限可回收和高危操作可追溯。
- 冲突处理原则：`角色权限以角色文档为准，接口定义以接口文档为准，页面交互以前端文档为准`。

## 术语统一表

| 术语 | 说明 |
| --- | --- |
| 业务类问题 | 账户、班级、试题、试卷、成绩等常规业务问题 |
| 考试类问题 | 考试分发、答题异常、监考告警等考试过程问题 |
| 系统类问题 | 服务故障、网络异常、资源不可用、备份恢复等平台问题 |
| 临时权限 | 带有效期的短期授权，到期自动回收 |
| 高危操作 | 成绩修改、数据恢复、日志导出、系统配置修改等需审批和审计的操作 |

## 一、设计原则

1. 最小权限：所有角色默认只能访问完成本职工作所需的最小数据和操作。
2. 审计分离：高危业务操作、系统配置、数据恢复和日志导出必须与普通业务权限分离。
3. 数据归属：教师只能处理本班或本人创建的数据，学生只能查看自身数据，管理员只能处理授权范围内的数据。
4. 类型分派：问题申报按问题类型分派处理人，避免业务问题、考试问题和系统故障混淆。
5. 临时可回收：临时授权必须设置有效期，到期自动回收，并保留审批记录。

## 二、角色定义与核心职责

| 角色分类 | 核心职责 | 适用人群 | 权限等级 |
| --- | --- | --- | --- |
| 超级管理员 | 角色模型维护、高危审批、系统配置、数据恢复、全局权限治理 | 系统负责人、平台主责人 | 最高 |
| 普通管理员 | 教师/学生账户管理、班级管理、试题审核、业务问题处理 | 学校教务、企业考试管理员 | 中高 |
| 教师 | 班级管理、试题试卷管理、考试组织、主观题批阅、考试类问题处理 | 授课教师 | 中等 |
| 学生 | 参加考试、查看成绩、入班申请、考试异常申报 | 学生、考生 | 最低 |
| 审计员 | 日志审计、成绩核查、异常行为复核、合规检查 | 内控、审计人员 | 高只读 |
| 系统运维 | 告警监控、系统日志分析、数据备份、系统类问题处理 | 运维工程师 | 运维专属 |

## 四、关键约束与审批链

### 4.1 数据范围约束

1. 教师仅能访问本班数据和本人创建内容。
2. 学生仅能访问自身账户、考试记录和成绩数据。
3. 普通管理员仅能访问教师/学生相关业务，不可越权操作超级管理员、审计员和运维核心权限。
4. 审计员为只读角色，不参与业务数据增删改。
5. 系统运维不参与考试业务处理，不得修改成绩、试题、试卷和角色配置。

### 4.2 高危操作审批链

1. 账户注销：学生注销自身账户需二次确认；管理员侧注销其他账户需超级管理员审批。
2. 成绩修改：发起人为超级管理员，审批人为另一名超级管理员，必须保留修改前后记录。
3. 日志导出：审计员导出日志需超级管理员审批。
4. 数据恢复：仅超级管理员可执行，恢复前必须先完成当前数据备份并保留审批记录。
5. 临时权限：由管理员或超级管理员审批，必须设置失效时间，最长 7 天。

### 4.3 数量与时间约束

1. 教师默认仅可创建 1 个专属班级。
2. 学生最多加入 3 个班级，强制班级不可自行退出。
3. 单次试题导入不超过 1000 题。
4. 登录失败 5 次后账户锁定 1 小时。
5. 成绩复批申请窗口为成绩发布后 7 天内。

## 五、安全规则（生产环境必落地）

1. 传输安全：所有 Web 接口和前后端通信统一使用 HTTPS。
2. 认证机制：统一采用 `accessToken + refreshToken` 登录态，不保存明文密码，不使用前端伪加密口径。
3. 鉴权模型：后端所有接口必须同时校验 `角色 + 资源 + 操作`，未授权请求直接返回 403。
4. 二次验证：系统配置修改、账户注销、日志导出、数据恢复、成绩修改等高危操作必须二次验证。
5. 审计留痕：敏感操作必须记录操作人、角色、时间、IP、请求 ID、操作对象、结果和审批信息，日志留存不少于 180 天。
6. 脱敏显示：手机号、身份证号、邮箱和导出数据按角色执行脱敏展示。
7. 文件安全：导入文件和截图必须校验格式、大小和恶意内容，禁止执行脚本或带宏文件。
8. 网络边界：超级管理员和系统运维的高危操作建议启用 IP 白名单或内网访问限制。

## 六、权限维护流程

1. 权限申请：申请人通过问题申报或审批流发起权限申请，注明原因、范围和有效期。
2. 权限审批：普通权限由普通管理员审批，高危权限由超级管理员审批。
3. 权限分配：审批通过后在系统管理模块中执行分配，并同步通知申请人。
4. 权限回收：临时权限到期自动回收，长期权限由审批人或超级管理员回收并记录原因。
5. 权限审计：审计员按月抽查权限分配、日志导出和高危审批记录。

## 七、扩展设计（非本期实现）

1. 家长角色：仅查看绑定学生的成绩和考试通知。
2. 多租户管理员：用于多校区、多企业隔离部署场景。
3. 应急模式：故障场景下为超级管理员提供临时旁路权限，但需强制审计。
4. 防作弊扩展：可追加多设备登录检测、人脸校验、摄像头监考等能力。
